# 本科开发类毕业论文审核

对开发类本科毕业论文进行多维度审核，生成审核报告并在Word中添加批注。

## 审核定位

本科论文审核标准：
- **重点关注**：结构完整性、逻辑自洽性、格式规范性、明显错误
- **适度关注**：内容一致性（需求-设计-实现-测试链路）、图表完整性
- **不苛求**：研究深度、创新性、文献综述广度
- **严格检查**：前后矛盾、数据不一致、明显抄袭/AI痕迹

## 审核规则体系

读取 [references/rules.md](references/rules.md)，包含四类规则：

| 类别 | 编号前缀 | 说明 |
|------|---------|------|
| 目录结构规则 | R1-R7 | 7章必选/可选节检查 |
| 内容一致性规则 | C1-C6 | 跨章节内容比对 |
| 格式规范规则 | F1-F6 | 图表、标题、标点、人称、参考文献、代码格式 |
| 内容逻辑规则 | L1-L5 | 前后逻辑、内容完整性、明显错误、抄袭痕迹 |
| 深度审核规则 | D1-D3 | 内容准确性、语言规范性、表述专业性（仅修改稿生效） |

## 审核工作流

### 第零步：审核准备

在开始审核前，需确认以下信息（如用户未提供则主动询问）：

1. **批注作者姓名**：Word批注中显示的审核人姓名
2. **待审核论文**：确认具体审核哪篇论文
3. **论文版本**：当前论文属于**初稿**还是**修改稿**？
   - **初稿**：仅检查基础规则（结构R、一致性C、格式F、逻辑L），不启用深度审核规则
   - **修改稿**：在基础规则之上，额外启用深度审核规则（D类规则），对内容准确性、语言规范性、表述专业性进行逐段审查

### 第一步：环境准备

**Python 虚拟环境**：在工作根目录创建项目本地 venv（如尚不存在）：
```bash
cd <工作根目录>
uv venv .venv && source .venv/bin/activate && uv pip install defusedxml lxml
```
后续所有 Python 命令均需先 `source <工作根目录>/.venv/bin/activate`。

### 第二步：目录管理

为每篇论文创建独立的审核工作目录，保持项目整洁：

```
<工作根目录>/
  .venv/                       # Python 虚拟环境（全项目共用）
  <论文名>/                    # 以论文文件名（去掉.docx后缀）命名目录
    <论文名>.docx              # 复制原始论文到此目录
    <论文名>.md                # pandoc 转换的 markdown
    <论文名>-unpacked/         # 解包后的文档目录
    <论文名>-审核报告.md       # 审核报告
    <论文名>-已审核.docx       # 添加批注后的最终文档
```

操作步骤：
1. 取论文文件名（去掉 .docx 后缀）作为目录名
2. 在工作根目录下创建该子目录
3. 将原始论文（docx）复制到该子目录中
4. 后续所有中间文件和输出文件均放在该子目录内

### 第三步：读取论文

如果用户尚未提供 markdown 文件，则转换：
```bash
pandoc "<论文路径>.docx" -o "<输出路径>.md"
```
**注意**：`.doc` 格式 pandoc 无法转换，需先用 libreoffice 或让用户转为 `.docx`。

读取 markdown，解析章节结构，记录论文标题、作者、系统名称、技术栈等关键信息。

**注意**：markdown 文件可能很大，需分段读取（每次200行左右）。先读目录部分掌握结构，再按需读取各章节内容。

### 第四步：逐维度检查

按以下顺序检查，对每条规则给出判定：
- **通过** — 符合要求
- **部分通过** — 存在但有瑕疵
- **不通过** — 缺失或严重不符
- **不适用** — [可选]规则且论文未涉及

**检查顺序**：

1. **结构检查（R）**：逐章验证必要章节是否存在、内容是否达到基本要求
2. **一致性检查（C）**：跨章节比对，重点检查系统名称统一、技术栈前后一致、需求→设计→实现→测试链路对应、图表引用正确、数量描述与实际条目匹配
3. **格式检查（F）**：图表标号标题、章节编号、标点符号、人称语气、参考文献格式和数量、代码呈现方式
4. **逻辑检查（L）**：前后矛盾、内容空洞、明显错别字和技术错误、残留链接/占位符/模板痕迹、复制粘贴痕迹
5. **深度审核（D）**[仅修改稿]：内容准确性（论点论据逻辑性）、语言规范性（病句/成分残缺/搭配不当）、表述专业性（口语化/主观化表达）

**特殊论文类型处理**：
- 游戏开发类论文（如Unity3D）：数据库设计(R4.4)标记为"不适用"，架构设计以模块设计代替
- 非Web系统论文：根据实际技术栈灵活调整规则适用性

### 第五步：生成审核报告

在论文目录下生成 `<论文名>-审核报告.md`：

```markdown
# 论文审核报告

**论文标题**：xxx
**作者**：xxx
**审核日期**：xxxx-xx-xx

## 一、结构检查结果（R）

| 规则编号 | 检查项 | 结果 | 说明 |
|---------|--------|------|------|

## 二、一致性检查结果（C）

| 规则编号 | 检查项 | 结果 | 说明 |
|---------|--------|------|------|

## 三、格式检查结果（F）

| 规则编号 | 检查项 | 结果 | 说明 |
|---------|--------|------|------|

## 四、逻辑检查结果（L）

| 规则编号 | 检查项 | 结果 | 说明 |
|---------|--------|------|------|

## 五、深度审核结果（D）[仅修改稿]

| 规则编号 | 检查项 | 结果 | 说明 |
|---------|--------|------|------|

> 如为初稿审核，此部分标注"初稿审核，未启用深度审核规则"。

## 六、问题汇总

按严重程度分类：
1. [必须修改] xxx — 结构缺失、前后矛盾等
2. [建议修改] xxx — 格式不规范、内容偏薄等
3. [可选改进] xxx — 锦上添花的建议

## 七、审核总结

总体评价及优先修改建议。
```

### 第六步：在Word中添加批注

使用 docx 技能在原始 Word 文档中添加批注。分为**解包→定位→批注→打包**四个子步骤。

#### 6.1 解包文档
```bash
source <工作根目录>/.venv/bin/activate
python3 /Users/awsling/.claude/plugins/marketplaces/anthropic-agent-skills/skills/docx/ooxml/scripts/unpack.py "<论文.docx>" "<解包目录>"
```

#### 6.2 关键词定位问题段落（两步法）

**第一步：全文段落索引导出**——遍历所有段落文本，输出 `p[索引]: 文本前120字`：
```bash
source <工作根目录>/.venv/bin/activate && python3 -c "
import defusedxml.minidom as minidom
doc = minidom.parse('<解包目录>/word/document.xml')
for i, p in enumerate(doc.getElementsByTagName('w:p')):
    runs = p.getElementsByTagName('w:r')
    full_text = ''
    for r in runs:
        texts = r.getElementsByTagName('w:t')
        full_text += ''.join(t.childNodes[0].data if t.childNodes else '' for t in texts)
    if full_text.strip():
        print(f'p[{i}]: {full_text[:120]}')
" 2>&1 | head -300
```

**第二步：按关键词精准搜索**——从审核报告中提取问题关键词，批量定位对应段落索引：
```bash
source <工作根目录>/.venv/bin/activate && python3 -c "
import defusedxml.minidom as minidom
doc = minidom.parse('<解包目录>/word/document.xml')
keywords = ['关键词1', '关键词2', '关键词3']  # 从审核报告提取的问题关键词
for i, p in enumerate(doc.getElementsByTagName('w:p')):
    runs = p.getElementsByTagName('w:r')
    full_text = ''
    for r in runs:
        texts = r.getElementsByTagName('w:t')
        full_text += ''.join(t.childNodes[0].data if t.childNodes else '' for t in texts)
    for kw in keywords:
        if kw in full_text:
            print(f'p[{i}] [{kw}]: {full_text[:150]}')
            break
"
```

**关键词选择原则**：选**短且唯一**的片段，如错别字本身（"登陆""折现"）、问题句子片段（"数据库架构的。"）、章节标题关键词等。

#### 6.3 批量添加批注

所有批注在**一个 Python 脚本**中完成，最后只调用**一次 save**：
```bash
source <工作根目录>/.venv/bin/activate && python3 << 'PYEOF'
import sys
sys.path.insert(0, '/Users/awsling/.claude/plugins/marketplaces/anthropic-agent-skills/skills/docx/')
from scripts.document import Document

doc = Document('<解包目录>', author="<审核人姓名的unicode转义>", initials="<姓名首字母缩写>")

def get_p(idx):
    return doc["word/document.xml"].dom.getElementsByTagName('w:p')[idx]

# 段落级批注（适用于结构缺失、章节问题等）
p = get_p(<段落索引>)
doc.add_comment(start=p, end=p, text="[R2.1] \u7f3a\u5c11\u72ec\u7acb\u7684\u6280\u672f\u7ae0\u8282...")

# 所有批注添加完毕后，只调用一次 save
doc.save(validate=False)
print("Done: N comments added")
PYEOF
```

**关键注意事项**：
- **必须使用 `validate=False`**：原始文档经常有格式问题，强制校验会导致保存失败
- **Python 脚本中所有中文必须用 unicode 转义**：如 `"\u738b\u660e"` 而非 `"王明"`，避免编码问题导致语法错误或 XML 解析失败
- **author 参数用 unicode**：如 `author="\u738b\u660e"` 表示"王明"
- **使用 heredoc `<< 'PYEOF'`**（带引号）防止 shell 对 `$` 等特殊字符做替换
- 每个问题对应一条批注，批注格式：`[规则编号] 问题描述和修改建议`

**批注粒度原则**：
- **R/C/L类规则**：段落级批注（`start=p, end=p`），指向问题所在段落
- **D类规则（语言/表述）**：可精确到 run 级别，但段落级也可接受
- 一个段落有多处不同问题时，分别添加独立批注


## 高频问题清单（12篇实战经验）

以下是审核12篇论文后总结的高频问题，按发现频率排序，审核时重点关注：

### 1. 复制粘贴错误（10/12篇）— 最高频
- 节标题与节内容不匹配（如标题写"商品管理"内容说的是"订单管理"）
- 实体描述套用模板未修改（如所有实体都写"培训实体图"）
- **从其他论文复制内容未修改**（如进销存系统的非功能需求中出现"住户""维修申请""维修人员"等物业系统内容）
- 结论大段复制绪论内容
- 中英文摘要关键词不一致（如中文"拍照预约"对应英文"Wedding photography"）
- **多个不同功能模块共用同一张截图**（如图5-15/5-17/5-19均引用image29.png）
- **检查方法**：逐节比对标题与首段内容是否匹配；检查图片文件名是否重复引用

### 2. 缺少独立的"相关技术与工具"章节（7/12篇）
- 技术介绍被混杂在"开发环境"或"需求分析"中，而非独立成章
- 实际使用的技术（如Vue、MyBatis、Redis等）在技术介绍章节中未提及
- **检查方法**：确认是否存在独立的第二章专门介绍技术栈

### 3. 章节编号错误（8/12篇）
- 子节编号与父节不匹配（如第4.3节下出现4.2.1而非4.3.1）
- 同一编号出现两次（如两个图5-21）
- 章节编号格式前后不统一（如前三章"1/2/3"后三章"第四章/第五章"）
- 标题编号与标题之间缺少空格（如"3.1可行性分析"）

### 4. 图表编号问题（8/12篇）
- 图/表编号重复（如多个图4-3、两个图5-21）
- 编号跳跃（如从图4.3直接跳到图4-7）
- 图题格式不统一（有的带"图"前缀有的不带，编号后有无空格不一致）
- "如图X所示"但图不存在或位于很远处
- 图片实为工具截图而非规范的UML图（如数据库控制台截图冒充E-R图）

### 5. 缺少关键章节/内容（7/12篇）
- 缺少非功能需求分析节（R3.3）
- 缺少核心业务流程设计节/顺序图/活动图（R4.3）
- 缺少系统架构设计节/架构图（R4.1）
- 缺少可行性分析节（R3.1）
- 缺少开发环境节（R5.1）
- 缺少规范的E-R图（R4.4）

### 6. 错别字和基本语言错误（10/12篇）
- "登陆"→"登录"（最常见）
- "折现"→"折线"、"景飒"→"竞赛"、"吸怪"→"修改"、"请过"→"情况"
- 句子不完整（如"完成了对数据库架构的。"缺宾语）
- 技术概念错误（如"RSA对称加密"应为"RSA非对称加密"）
- 成语用错（如"功半事倍"应为"事半功倍"）
- "的/地"混用（如"有效的提升"应为"有效地提升"）

### 7. 技术描述矛盾（5/12篇）
- 绪论提到的技术（如SpringSecurity、MyBatis-Plus）第二章未介绍
- 架构图中包含未实际使用的技术（如Node.js）
- 结论中技术栈描述省略版本号，与标题不一致
- 实现描述夸大（如本科项目声称使用"分库分表""OSS切片上传""Redis分布式锁"）

### 8. 残留痕迹（3/12篇）
- ABSTRACT标题含百度链接
- 图片文件名暴露非作者姓名（如"曾续缘ER图"但作者是"欧阳光航"）
- 截图路径暴露QQ号等个人信息
- **检查方法**：关注 markdown 中的超链接、图片文件名和 Word rels 中的链接

### 9. 论文结构描述与实际不一致（5/12篇）
- 1.4节"论文组织结构"中描述的章节名称/内容与实际目录不匹配
- 声称某章包含某内容但实际缺失（如称第3章有"系统原型设计"但实际无此节）

### 10. 其他常见问题
- 系统名称全文不统一（"平台""系统""网站"混用）
- 使用"我们"而非"本文/本系统"（人称不统一）
- 参考文献格式不统一（部分缺[J]/[M]标注）
- 参考文献与论文主题无关（如拍照类文献出现在预约管理系统论文中）
- 代码以截图而非文本形式展示
- 结论章缺少独立的"展望"小节
- 口语化/主观化表述（"不错的""比较能够""很好的""大大提升"）
- 测试数据前后矛盾（声称164个测试用例但实际仅展示20个）

## 批量审核策略（多篇论文）

### 推荐流程：主线程统一调度

经实战验证，后台 agent 经常因权限问题无法执行 Bash 命令。**推荐的批量审核流程**：

1. **统一准备**（主线程）：
   - 创建 venv、安装依赖
   - 批量创建子目录、复制文件
   - 批量 pandoc 转换 markdown
   - 批量 unpack 解包 docx

2. **并行审核报告**（可用 agent）：
   - 阶段一（仅读取+分析+写报告）不需要 Bash，agent 可以用 Read + Write 工具完成
   - 为每篇论文启动一个 agent，只负责：读取 md → 逐维度检查 → 生成审核报告
   - agent prompt 中提供完整的规则列表和论文 md 路径

3. **主线程逐篇批注**（串行执行）：
   - 读取每篇审核报告，提取问题清单
   - 关键词搜索定位段落索引
   - 编写并执行批注 Python 脚本
   - pack 打包
   - 每篇约需 3 轮 Bash 调用（搜索→批注→打包）

4. **统一打包交付**：
   ```bash
   mkdir -p 审核结果汇总
   for name in "论文1" "论文2"; do
     mkdir -p "审核结果汇总/${name}"
     cp "${name}/${name}-已审核.docx" "审核结果汇总/${name}/"
     cp "${name}/${name}-审核报告.md" "审核结果汇总/${name}/"
   done
   zip -r "审核结果汇总.zip" "审核结果汇总/"
   ```

### 效率要点

- **文件名含特殊字符时**（`+`、空格、中文括号`（）`、英文括号`()`），所有路径必须用双引号包裹
- agent prompt 中必须提供**精确的完整路径**，不要让 agent 自行拼接
- 批量操作尽量用 for 循环一次性完成（如批量 mkdir、cp、unpack）
- 审核报告生成是最耗时的部分（需逐段阅读），批注添加相对快速

## 审核注意事项

- [可选但建议]项（R3.4, R4.5, R6.3）缺失时标注"建议补充"而非"不通过"
- 本科论文不必苛求：参考文献数量（8篇以上即可）、研究现状深度、创新点
- 但以下问题必须严格指出：缺少必选章节、图表引用不存在、前后数据矛盾、残留百度链接等明显痕迹
- 批注语气应具有建设性，指出问题的同时给出修改方向
- 游戏开发、嵌入式等非Web系统论文，灵活调整数据库设计等规则的适用性
